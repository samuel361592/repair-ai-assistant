# LangChain 維修問題分析助手 v0.3 RAG Manual QA 規格

## 1. 版本目標

本版本目標是在 v0.2 Structured Output 的基礎上，新增「維修手冊 RAG 問答」能力。

v0.2 已完成：

```text
使用者輸入故障描述
→ ChatPromptTemplate
→ ChatOpenAI Structured Output
→ RepairAnalysis 結構化物件
→ CLI 格式化成 Markdown 顯示
```

v0.3 需要新增：

```text
本地維修手冊文件
→ 文件載入
→ 文件切片
→ Embedding
→ Vector Store
→ Retriever
→ 使用者提問
→ 檢索相關段落
→ 根據檢索內容回答
→ 顯示來源
```

本版本的目標不是取代 v0.2，而是新增一個獨立的 RAG 問答功能。

---

## 2. 為什麼要做 v0.3 RAG

v0.2 的維修分析主要依賴模型本身的推理能力，適合做初步問題整理，但缺少外部知識依據。

例如使用者問：

```text
TASKalfa 2553ci 出現 C6600 要怎麼處理？
```

如果沒有維修手冊，模型可能只能用一般經驗回答，容易出現：

* 回答不夠精準
* 無法引用來源
* 不知道特定機型錯誤碼
* 容易產生幻覺
* 無法確認答案是否真的來自文件

v0.3 要透過 RAG 讓系統根據本地維修手冊內容回答。

核心原則：

```text
模型不能只靠記憶回答
回答必須根據檢索到的文件段落
資料不足時要明確說資料不足
回答需要顯示來源
```

## 3. 本版本範圍

### 3.1 本版本要做

v0.3 需要完成：

1. 建立本地手冊資料夾
2. 支援讀取 `.md` 或 `.txt` 維修手冊
3. 將文件切成 chunks
4. 使用 Embedding 建立向量資料
5. 使用 Chroma 作為本地 vector store
6. 支援建立或重建知識庫
7. 支援使用者輸入問題
8. 從 vector store 檢索相關段落
9. 根據檢索段落產生回答
10. 回答需顯示來源資訊
11. 保留 v0.2 原本 CLI，不可破壞既有功能

---

### 3.2 本版本不做

v0.3 不處理以下功能：

* PDF 解析
* 掃描 PDF OCR
* 表格解析
* 圖片解析
* 多模態文件理解
* CRM API 串接
* 歷史維修紀錄查詢
* Agent
* Tool Calling
* LangGraph
* FastAPI
* Web UI
* 權限控管
* 使用者登入
* 雲端向量資料庫

PDF 讀取可放到 v0.3.1 或 v0.4 之後處理。

## 5. 新增資料夾與檔案

### 5.1 data/manuals

新增：

```text
data/manuals/
```

用來放測試用維修手冊。

本版本先使用 Markdown 或純文字文件，例如：

```text
data/manuals/sample_manual.md
```

範例內容：

```md
# 印表機維修手冊範例

## 卡紙問題

當設備發生卡紙時，請先確認卡紙位置。常見位置包含進紙匣、轉印區、定影區與出紙口。

### 出紙口卡紙

若紙張卡在出紙口附近，可能原因包含：
1. 出紙口有殘紙或異物
2. 出紙滾輪髒污或磨損
3. 紙張受潮或規格不符合設備設定

建議處理步驟：
1. 關閉設備電源
2. 打開出紙口蓋板
3. 小心取出殘紙
4. 檢查出紙滾輪是否髒污
5. 確認紙張尺寸與紙匣設定一致

## 異音問題

設備運轉時出現異音，可能與齒輪、滾輪、風扇或紙張傳送機構有關。

建議先確認異音發生位置，再判斷是否需要拆機檢查。
```

## 8. 功能需求

### FR-001：建立 RAG 文件載入功能

新增檔案：

```text
src/repair_assistant/rag.py
```

需要支援讀取：

```text
data/manuals/*.md
data/manuals/*.txt
```

每份文件需要轉成 LangChain `Document`。

Document metadata 至少包含：

```text
source
filename
```

如果能解析標題，可以額外包含：

```text
section
```

---

### FR-002：建立文件切片功能

需要使用 text splitter 將文件切成 chunks。

建議參數：

```python
chunk_size = 800
chunk_overlap = 120
```

每個 chunk metadata 需要保留：

```text
source
filename
chunk_index
```

---

### FR-003：建立 Embedding

使用 `OpenAIEmbeddings` 建立 embedding model。

設定來源：

```text
OPENAI_API_KEY
OPENAI_BASE_URL
OPENAI_EMBEDDING_MODEL
```

若 `OPENAI_EMBEDDING_MODEL` 未設定，預設使用：

```text
text-embedding-3-small
```

---

### FR-004：建立 Chroma Vector Store

使用 Chroma 作為本地 vector store。

儲存位置：

```text
data/vectorstore/chroma
```

需支援重建 vector store。

---

### FR-005：建立建庫指令

新增 CLI 指令：

```text
repair-rag-ingest
```

執行後會：

```text
讀取 data/manuals
→ 切片
→ 建立 embedding
→ 寫入 Chroma vector store
```

建議在 `pyproject.toml` 新增 script：

```toml
[project.scripts]
repair-assistant = "repair_assistant.main:main"
repair-rag-ingest = "repair_assistant.rag_cli:ingest_main"
repair-rag-qa = "repair_assistant.rag_cli:qa_main"
```

---

### FR-006：建立 RAG 問答指令

新增 CLI 指令：

```text
repair-rag-qa
```

執行後會提示使用者輸入維修問題。

使用者輸入例如：

```text
印表機出紙口卡紙要怎麼處理？
```

系統需要：

1. 從 Chroma vector store 檢索相關 chunks
2. 將 chunks 放入 prompt context
3. 呼叫 ChatOpenAI
4. 根據 context 回答
5. 顯示來源

---

### FR-007：回答需要根據文件內容

Prompt 必須要求模型：

1. 只能根據提供的文件內容回答
2. 若文件內容不足，需回答「目前手冊資料不足，無法確認」
3. 不可編造手冊沒有提到的錯誤碼、步驟或零件
4. 回答需使用繁體中文
5. 回答需包含可執行步驟

---

### FR-008：回答需顯示來源

回答後需顯示來源，例如：

```md
## 回答

根據手冊內容，若紙張卡在出紙口附近，建議先關閉設備電源，打開出紙口蓋板，小心取出殘紙，並檢查出紙滾輪是否髒污或磨損。

## 來源

1. data/manuals/sample_manual.md，chunk 2
2. data/manuals/sample_manual.md，chunk 3
```

來源資訊應由程式根據 retrieved documents metadata 產生，不應完全交給模型自由生成。

---


或：


v0.3 新增 RAG 功能時，不得破壞原本 `RepairAnalysis` structured output。

## 9. 非功能需求

### NFR-001：RAG 與 v0.2 分離

v0.3 新增 RAG 功能，但不應把 v0.2 的 `analyze_repair_problem()` 改成 RAG。

v0.2 負責：

```text
故障描述 → 結構化分析
```

v0.3 新增：

```text
維修手冊問題 → 文件檢索 → 根據手冊回答
```

兩者可以共存，但不要混在一起。

---

### NFR-002：不要提交本地向量庫

以下資料夾不可提交：

```text
data/vectorstore/
```

---

### NFR-003：測試需可離線執行

單元測試不得依賴實際 OpenAI API。

需要用 mock 測：

* 文件讀取
* 文件切片
* source formatting
* empty question validation
* no retrieved documents handling

端到端 RAG 建庫與問答可以作為手動測試。

---

### NFR-004：錯誤訊息清楚

以下情況需有清楚錯誤訊息：

1. `data/manuals` 不存在
2. 沒有任何 `.md` 或 `.txt` 文件
3. embedding API 呼叫失敗
4. vector store 尚未建立
5. 使用者輸入空白問題
6. retriever 找不到相關內容

```text
你是一位謹慎的維修手冊問答助理。

請只根據下方提供的手冊內容回答使用者問題。

規則：
1. 只能使用手冊內容回答。
2. 如果手冊內容不足，請回答：「目前手冊資料不足，無法確認。」
3. 不要編造手冊沒有提到的錯誤碼、零件、步驟或原因。
4. 回答請使用繁體中文。
5. 若手冊有處理步驟，請整理成條列式。
6. 若涉及安全風險，請提醒停止操作並交由合格維修人員處理。

手冊內容：
{context}

使用者問題：
{question}
```
