# LangChain 維修問題分析助手 v0.2 Structured Output 規格

## 1. 版本目標

本版本目標是在 v0.1 的基礎上，將維修問題分析結果從「純 Markdown 文字輸出」升級為「結構化資料輸出」。

v0.1 已完成以下流程：

```text
使用者輸入故障描述
→ PromptTemplate
→ ChatOpenAI
→ StrOutputParser
→ Markdown 文字輸出
```

v0.2 需要調整為以下流程：

```text
使用者輸入故障描述
→ PromptTemplate
→ ChatOpenAI Structured Output
→ RepairAnalysis 結構化物件
→ CLI 格式化成 Markdown 顯示
```

本版本的重點是讓模型輸出可以被程式穩定處理，而不是只產生給人看的文字。

本版本不新增以下功能：

* RAG
* Agent
* Tool Calling
* LangGraph
* PDF 讀取
* Embedding
* Vector Store
* Retriever
* FastAPI
* Web UI
* CRM API 串接

---

## 2. 為什麼要做 Structured Output

v0.1 的輸出是純 Markdown 文字，適合人類閱讀，但不適合程式後續處理。

例如 v0.1 可能輸出：

```md
## 問題摘要

印表機在出紙口附近反覆卡紙。

## 可能原因

1. 出紙口可能有殘紙或異物
2. 滾輪可能髒污或磨損
3. 紙張規格可能不符合設定
```

這種格式雖然可讀，但程式很難穩定取得：

* 問題摘要
* 可能原因清單
* 檢查步驟清單
* 補問問題清單
* 優先處理步驟清單

v0.2 要將輸出改成結構化物件，例如：

```python
RepairAnalysis(
    summary="印表機在出紙口附近反覆卡紙。",
    possible_causes=[
        "出紙口可能有殘紙或異物阻塞",
        "出紙滾輪可能髒污、磨損或打滑",
        "紙張規格、紙張品質或紙匣設定可能不一致",
    ],
    check_steps=[
        "檢查出紙口是否有殘紙、紙屑或異物",
        "檢查出紙滾輪是否髒污、磨損或無法正常帶紙",
        "確認紙張尺寸、厚度與紙匣設定是否一致",
    ],
    questions=[
        "卡紙是否每次都發生在同一個位置？",
        "目前使用的紙張尺寸、厚度與種類是什麼？",
        "最近是否更換過紙張、耗材或清潔過機器？",
    ],
    priority_steps=[
        "先檢查出紙口是否有明顯殘紙或異物",
        "再確認紙張與紙匣設定是否正確",
        "最後檢查滾輪與出紙機構是否異常",
    ],
)
```

本版本的核心原則是：

```text
AI 負責分析內容
程式負責資料結構與顯示格式
```

這樣未來才能更容易接：

* FastAPI API 回傳
* Web UI 顯示
* CRM 欄位對應
* 資料庫儲存
* RAG 引用來源
* Agent Tool Calling
* LangGraph workflow 狀態傳遞

## 3. 功能需求

### FR-001：新增 RepairAnalysis Schema

新增檔案：

```text
src/repair_assistant/schemas.py
```

在此檔案中建立 Pydantic schema：

```python
from pydantic import BaseModel, Field


class RepairAnalysis(BaseModel):
    summary: str = Field(description="一句話整理使用者描述的故障問題")
    possible_causes: list[str] = Field(description="可能原因，至少 3 筆")
    check_steps: list[str] = Field(description="初步檢查方向，至少 3 筆")
    questions: list[str] = Field(description="需要補問的資訊，至少 3 筆")
    priority_steps: list[str] = Field(description="建議處理優先順序，至少 3 筆")
```

欄位說明：

| 欄位              | 型別        | 說明              |
| --------------- | --------- | --------------- |
| summary         | str       | 一句話整理故障問題       |
| possible_causes | list[str] | 可能原因，至少 3 筆     |
| check_steps     | list[str] | 初步檢查方向，至少 3 筆   |
| questions       | list[str] | 需要補問的資訊，至少 3 筆  |
| priority_steps  | list[str] | 建議處理優先順序，至少 3 筆 |

---

### FR-002：使用 LangChain Structured Output

`chains.py` 需要從原本的文字輸出：

```python
chain = prompt | model | parser
```

調整為使用 LangChain structured output 能力。

目標概念：

```python
structured_model = model.with_structured_output(RepairAnalysis)
chain = prompt | structured_model
```

不得只靠 prompt 要求模型輸出 JSON。

也就是說，本版本不接受以下做法作為主要實作方式：

```text
請輸出 JSON 格式
```

然後再手動 `json.loads()`。

本版本應使用 LangChain 提供的 structured output 機制，讓模型輸出符合 `RepairAnalysis` schema。

## 5. 設計說明

### 5.1 schemas.py 設計

`schemas.py` 負責定義結構化輸出格式。

建議內容：

```python
from pydantic import BaseModel, Field


class RepairAnalysis(BaseModel):
    summary: str = Field(description="一句話整理使用者描述的故障問題")
    possible_causes: list[str] = Field(description="可能原因，至少 3 筆")
    check_steps: list[str] = Field(description="初步檢查方向，至少 3 筆")
    questions: list[str] = Field(description="需要補問的資訊，至少 3 筆")
    priority_steps: list[str] = Field(description="建議處理優先順序，至少 3 筆")
```

若要更嚴格，可以額外加入欄位說明，但本版本不強制做數量驗證器。

---

### 5.2 chains.py 設計

`chains.py` 負責：

1. 建立模型
2. 建立 Prompt
3. 套用 structured output
4. 回傳 `RepairAnalysis`

建議流程：

```text
get_settings()
→ ChatOpenAI()
→ ChatPromptTemplate
→ model.with_structured_output(RepairAnalysis)
→ prompt | structured_model
→ chain.invoke()
→ RepairAnalysis
```

概念程式碼：

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

from src.repair_assistant.config import get_settings
from src.repair_assistant.schemas import RepairAnalysis


def build_repair_analysis_chain():
    settings = get_settings()

    model = ChatOpenAI(
        model=settings["model"],
        base_url=settings["base_url"],
        api_key=settings["api_key"],
        temperature=0,
    )

    prompt = ChatPromptTemplate.from_template("""
你是一位印表機與設備維修助理。

請根據使用者提供的故障描述，產生結構化維修分析。

要求：
1. 不要假裝已經知道真正原因。
2. 可能原因需使用「可能」語氣。
3. 檢查方向需具體可執行。
4. 補問資訊需能協助維修人員縮小問題範圍。
5. 建議處理優先順序需由低風險、易檢查項目開始。
6. 每個 list 欄位至少產生 3 筆內容。

使用者故障描述：
{problem}
""")

    structured_model = model.with_structured_output(RepairAnalysis)

    return prompt | structured_model


def analyze_repair_problem(problem: str) -> RepairAnalysis:
    chain = build_repair_analysis_chain()
    return chain.invoke({"problem": problem})
```

實作時可以依照目前專案既有風格調整，但需符合 v0.2 需求。

概念程式碼：

```python
from src.repair_assistant.schemas import RepairAnalysis


def format_numbered_list(items: list[str]) -> str:
    return "\n".join(f"{index}. {item}" for index, item in enumerate(items, start=1))


def format_repair_analysis(analysis: RepairAnalysis) -> str:
    return f"""## 問題摘要

{analysis.summary}

## 可能原因

{format_numbered_list(analysis.possible_causes)}

## 初步檢查方向

{format_numbered_list(analysis.check_steps)}

## 需要補問的資訊

{format_numbered_list(analysis.questions)}

## 建議處理優先順序

{format_numbered_list(analysis.priority_steps)}
"""
```

## 6. Prompt 調整要求

Prompt 需明確要求模型：

1. 根據使用者提供的故障描述分析
2. 不要假裝已經知道真正原因
3. 不要直接宣稱某個原因一定正確
4. 可能原因需使用「可能」語氣
5. 檢查方向需具體可執行
6. 補問資訊需能協助維修人員縮小問題範圍
7. 建議處理優先順序需由低風險、易檢查項目開始
8. 每個 list 欄位至少產生 3 筆內容

範例 Prompt 方向：

```text
你是一位印表機與設備維修助理。

請根據使用者提供的故障描述，產生結構化維修分析。

注意：
- 不要假裝已經知道真正原因。
- 可能原因請使用「可能」語氣。
- 檢查方向需具體可執行。
- 需要補問的資訊需能幫助維修人員縮小問題範圍。
- 建議處理優先順序需由低風險、易檢查項目開始。
- 每個 list 欄位至少產生 3 筆。

使用者故障描述：
{problem}
```
